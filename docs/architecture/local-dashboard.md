# Local Dashboard

Local Dashboard v2 is a read-only multi-page Control Room for Dev Integrity.

## Purpose

The dashboard visualizes current operational trust state from generated local artifacts. It helps humans inspect recent reports, Control Room status, release gate score, active workflows, GitHub summary fields, and timeline summary without opening every markdown file manually.

The dashboard is intentionally operational, not SaaS and not auth. It is a local visualization layer over existing Integrity outputs.

## Local Read-Only Artifact Viewer

The dashboard is plain HTML, CSS, and JavaScript under `tools/dev-integrity-dashboard/`. It has no build step and no framework.

It is served locally with:

```bash
npm run dashboard
```

## Multi-Page Control Room Layout

The dashboard is split into focused static pages:

- Overview: primary Control Room decision cards and a short set of blockers, warnings, active workflows, and latest review metadata.
- Reviews: recent catalog entries with repo, generated time, trust, decision, workflow priority, release gate score, and markdown report links.
- Release Gates: release decision, confidence band, positive and negative contributors, required gate evidence, release workflow status, and release signal summary.
- Workflows: active workflows, repeated workflow patterns, workflow reasons, evidence needs, and recommended workflow path.
- Timeline: timeline summary markdown plus extracted confidence trend, repeated blockers, repeated workflows, and repeated risk drivers.
- GitHub: Git context, branch comparison, GitHub checks, GitHub Actions summary, failed and pending counts, and release signal linkage.

The pages share one static navigation bar. `shared.js` owns artifact loading and common formatting helpers. `dashboard.js` owns page rendering.

## Dashboard Philosophy

The Control Room prioritizes operational readability over metric density. Each page keeps the primary decision near the top, moves secondary evidence into panels or expandable sections, and links back to generated markdown for detailed review.

The dashboard does not recalculate Integrity outcomes. It displays fields and summaries produced by the review runner and timeline artifacts.

## Release Confidence Guidance

Overview and Release Gates include explanatory scoring guidance for release confidence. This is dashboard-only interpretation text. It clarifies that a `0%` confidence score means the change is not release-safe under the current scoring posture, not that the code has no value.

The dashboard currently describes three conceptual modes:

- Local Dev: early experimentation where missing CI, runtime, or release evidence should be softer.
- Staging: pre-release review where evidence, checks, workflow routing, and release signals matter more.
- Production Strict: release governance where red Control Room state, blocked decisions, recovery risk, failed checks, and missing evidence can collapse confidence quickly.

These mode descriptions are conceptual for now. They do not switch scoring modes, alter generated artifacts, recalculate confidence, or duplicate engine scoring logic.

## Drill-Down UX

The Overview page answers whether the current state is safe to proceed. Drill-down pages then separate the supporting evidence:

- Reviews for report history.
- Release Gates for release readiness.
- Workflows for operational routing.
- Timeline for repeated patterns.
- GitHub for cataloged source control and CI context.

Recent report rows open the corresponding generated markdown report. Expandable sections keep repeated evidence available without forcing it onto the first screen.

## Data Sources

The dashboard reads:

- `reports/catalog.json`
- `reports/timeline-summary.md`
- latest report markdown linked from the catalog

These are generated by the existing review runner.

## No New Logic

The dashboard does not duplicate engine logic. It does not recalculate risk, release gate scores, workflow priority, Control Room status, or timelines. It only displays existing fields from generated artifacts.

Timeline page summaries are extracted from headings in `reports/timeline-summary.md`. This is presentation parsing only; the dashboard does not derive new risk decisions from raw repo data.

## Safety

The dashboard does not:

- call external APIs
- write to GitHub
- deploy
- modify target repositories
- store secrets
- read raw logs
- use a database

## Future Dashboard Path

Future versions may add hosted Control Room surfaces, richer report drawers, charts over report history, and operator collaboration features. Hosted direction should keep the Integrity Engine and generated artifacts as the source of truth, with explicit boundaries for storage, authentication, tenancy, and external integrations.

## Limitations

- v2 requires local generated artifacts to exist.
- v2 uses browser `fetch()`, so it should be served over a local static server.
- v2 does not parse full markdown reports beyond linking to them.
- v2 has no authentication because it is local-only.
