# Social Media Rules

AuthToolkit social graphics should look like infrastructure communication, not startup noise.

The goal is to make operational ideas easy to understand in a feed:

- release risk
- recovery posture
- workflow orchestration
- authentication systems
- API execution
- event delivery
- agent review

## Social Design Principles

- strong navy headline
- green accent for action or safe path
- lots of whitespace
- one concept per card
- minimal diagram when useful
- no busy backgrounds
- no fake screenshots
- no meme visual language for product announcements

## Standard Formats

Recommended sizes:

- Square: `1080 x 1080`
- Wide: `1600 x 900`
- LinkedIn document slide: `1080 x 1350`
- Open Graph card: `1200 x 630`

## Social Card Structure

Use this hierarchy:

```text
Eyebrow: AUTH TOOLKIT / RELEASE CONTROL / API INTAKE
Headline: short operational statement
Diagram or proof point
Footer: AuthToolkit mark + URL or product name
```

## Copy Style

Good:

- `AI-generated PRs still need release confidence.`
- `Passing tests are not the same as release readiness.`
- `Dry run first. Execute locally. Review with evidence.`
- `Recovery evidence belongs in the release path.`

Avoid:

- `10x your AI engineering with magic agents`
- `The future of AGI coding is here`
- `Deploy instantly with autonomous intelligence`

## Visual Motifs

Use:

- connected nodes
- release gate cards
- API request/response blocks
- webhook arrows
- dashboard status strips
- recovery flow timelines
- payment retry paths

Avoid:

- robot mascots
- neon gradients
- crypto grids
- abstract 3D blobs
- random icons without system meaning

## Color Usage

Default social card:

- background: white or soft mint
- headline: navy `#041B4D`
- accent: green `#7ED321`
- supporting text: muted slate
- lines: pale green or light gray

Dark social card:

- background: navy `#041B4D`
- headline: white
- accent: green
- line work: pale blue or soft mint

Use dark cards for major launches and strong statements. Use light cards for education and workflow explanations.

## Carousel Pattern

A carousel should teach one operational idea.

Example:

```text
Slide 1: Passing tests are not release confidence.
Slide 2: What can still be missing?
Slide 3: Evidence, rollback, owner confirmation.
Slide 4: Dev Integrity makes the release posture visible.
Slide 5: Run local-first. Review before merge.
```

## Product Screenshot Rules

If using dashboard screenshots:

- crop to one focused panel or workflow
- keep text readable
- do not show secrets, tokens, private repo names, or customer data
- add a simple label, not a marketing paragraph

## Voice

Social tone should be calm and precise.

Use:

- declarative statements
- short explanations
- operational language
- one clear CTA

Avoid:

- hype threads
- vague founder platitudes
- excessive AI vocabulary
- fearmongering
